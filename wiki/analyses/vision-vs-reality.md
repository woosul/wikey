# Karpathy 비전 vs 커뮤니티 현실

> 쿼리: "Karpathy 원문의 비전과 커뮤니티 현실 반응을 비교해줘"
> 근거: [[source-llm-wiki-gist]], [[source-llm-wiki-community]]

## 비교 매트릭스

| 주제 | Karpathy 비전 | 커뮤니티 현실 |
|------|-------------|-------------|
| **LLM 역할** | "LLM이 모든 부기 작업을 한다" | "claude.md 하나도 유지 못한다" (HN) |
| **유지보수 비용** | "거의 제로" | 새로운 기술 부채 — "뇌의 일부가 비어 있는 느낌" |
| **확장성** | index.md로 수백 페이지 가능 | "AI가 커지는 위키를 감당할 수 있는가?" |
| **RAG 대비** | "복리 성장 산출물" | "결국 RAG 아닌가?" 논쟁 |
| **미래** | 모듈식, 각자 도메인에 맞게 | "10M 컨텍스트면 불필요" vs "기다리면 아무것도 못 만듦" |
| **사용자 역할** | 소스 큐레이션 + 좋은 질문 | 깊이 사고 능력 상실 우려 |
| **사고 가치** | (언급 없음) | "문서 작성의 진짜 가치는 사고 정리 과정" |

## 합의된 영역

원문과 커뮤니티가 **동의하는 지점**:

1. **[[knowledge-compounding|탐색의 축적]]이 핵심 가치** — [[llmbase]]와 [[farzapedia]] 모두 확인
2. **RAG보다 에이전트 직접 탐색이 효과적** — Farzapedia: RAG → 파일 시스템 전환 후 효과 극대화
3. **세션 간 컨텍스트 영속성이 가장 큰 문제** — .brain 폴더, 학습 디렉토리 등 다양한 변형으로 동의
4. **[[byoai|BYOAI]] 원칙의 중요성** — 특정 모델 종속 회피

## 비전과 현실의 간극

Karpathy 원문은 의도적으로 **낙관적이고 추상적**이다 ("이 문서는 의도적으로 추상적이다. 패턴을 전달하는 것이 유일한 역할"). 구체적 실패 모드는 논의하지 않는다.

커뮤니티는 **실사용 경험**에서 간극을 발견:
- 비전: LLM이 일관적으로 위키를 유지 → 현실: LLM의 일관성 한계
- 비전: 유지보수 비용 제로 → 현실: 새로운 형태의 인지 비용 발생
- 비전: 사용자는 "좋은 질문"에 집중 → 현실: 그 과정에서 깊은 사고 기회를 잃을 수 있음

## 시사점

Karpathy의 비전은 **방향으로서 유효**하지만, 현실 구현에서는 커뮤니티가 지적한 **완화책이 필수**:
- LLM 일관성 → `validate-wiki.sh` + BYOAI 교차 검증
- 사고 위임 → 인제스트 시 사용자 논의 단계 필수화
- 확장성 → [[qmd]] 기반 다층 검색
