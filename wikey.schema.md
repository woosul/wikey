# Wikey Schema — LLM Wiki 마스터 스키마

> 프로바이더 독립 마스터 스키마. LLM 에이전트는 이 파일을 **먼저 읽고** 위키의 구조, 컨벤션, 워크플로우를 파악한 후 작업한다.
> 프로바이더별 도구 사용법은 각 설정 파일을 참조: `CLAUDE.md` (Claude Code), `AGENTS.md` (Codex), `local-llm/system-prompt.md` (로컬 LLM).

## 프로젝트 개요

Andrej Karpathy의 [LLM Wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) 패턴을 기반으로 한 개인 지식 베이스.

LLM이 원시 소스를 읽고, 구조화된 위키를 점진적으로 구축·유지하는 시스템. RAG와 달리 지식이 매 쿼리마다 재도출되지 않고 **영구적으로 축적**된다.

## 시스템 아키텍처

```
┌─────────────────────────────────────────────────────────────┐
│                    wikey/ = Obsidian 볼트                    │
│         (프로젝트 폴더 = Obsidian 볼트 = Git 저장소)          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌──────────┐    인제스트     ┌──────────────────────────┐  │
│  │  raw/    │ ──────────────> │  LLM 에이전트             │  │
│  │ (원시소스) │   쿼리/린트    │  - 소스 읽기·분석         │  │
│  └──────────┘ <───────────── │  - 위키 페이지 생성·수정   │  │
│       ▲                      │  - 교차참조·인덱스 관리    │  │
│       │ 사용자가              │  - 모순 감지·린트          │  │
│       │ 소스 추가             └───────────┬──────────────┘  │
│       │                                   │ 읽기/쓰기       │
│  ┌────┴─────┐                    ┌────────▼─────────┐      │
│  │ 사용자    │                    │  wiki/           │      │
│  │ (큐레이터) │ ◄── Obsidian ──── │  (마크다운 위키)   │      │
│  │          │    으로 브라우징     │  index.md        │      │
│  └──────────┘                    │  log.md          │      │
│       │                          │  entities/       │      │
│       │ 질문/지시                  │  concepts/       │      │
│       ▼                          │  sources/        │      │
│  ┌──────────┐                    │  analyses/       │      │
│  │ LLM     │                    └──────────────────┘      │
│  │ 세션     │                             │                │
│  └──────────┘                     Obsidian CLI로도 접근      │
│                                                             │
├─────────────────────────────────────────────────────────────┤
│  .obsidian/         Obsidian 설정 (그래프뷰, 플러그인, 테마)  │
│  wikey.schema.md    마스터 스키마 (이 파일)                   │
│  CLAUDE.md          Claude Code 프로바이더 설정               │
│  AGENTS.md          Codex 프로바이더 설정                     │
│  local-llm/         로컬 LLM 설정                            │
└─────────────────────────────────────────────────────────────┘
```

### 역할 분담

| 구성요소 | 역할 | 소유권 |
|---------|------|--------|
| **사용자** | 소스 큐레이션, 질문, 분석 방향 지시, 최종 판단 | raw/에 소스 추가, 쿼리/린트 지시 |
| **LLM 에이전트** | 위키 작성·유지의 모든 실행 작업 | wiki/ 전체 읽기/쓰기 |
| **Obsidian** | 위키 브라우징·시각화 인터페이스 | 읽기 전용 뷰어 + 그래프 뷰 |
| **Obsidian CLI** | LLM ↔ Obsidian 간 프로그래매틱 접근 | 검색, 백링크, 속성 조회 |
| **Git** | 버전 관리, 변경 이력, 롤백, 원격 백업 | 모든 파일의 이력 |
| **wikey.schema.md** | LLM의 행동 규칙·컨벤션·워크플로우 정의 | 사용자 + LLM 공동 진화 |

### Obsidian의 역할

Obsidian은 **코드 에디터에 비유하면 IDE**이고, LLM이 프로그래머이고, 위키가 코드베이스다.

| 기능 | 용도 |
|------|------|
| **파일 브라우저** | 위키 페이지 탐색, 디렉토리 구조 확인 |
| **마크다운 렌더링** | 위키 페이지 읽기 (프론트매터, 위키링크, 콜아웃) |
| **그래프 뷰** | 위키 전체 연결 구조 시각화 — 허브, 고아, 클러스터 파악 |
| **백링크 패널** | 특정 페이지를 참조하는 모든 페이지 확인 |
| **검색** | 전체 텍스트 검색 (GUI) + CLI 검색 |
| **Web Clipper** | 웹 기사 → 마크다운 변환 → `raw/articles/`에 저장 |
| **Bases** | 프론트매터 기반 메타데이터 대시보드 (테이블, 필터) |
| **CLI** | 터미널에서 볼트 조작 — LLM이 프로그래매틱하게 검색·읽기·속성 관리 |

**Obsidian은 위키를 직접 편집하지 않는다.** 위키 페이지의 생성·수정은 LLM 에이전트가 담당. Obsidian은 결과를 실시간으로 보여주는 뷰어이자, 그래프 뷰·백링크 등 LLM이 제공하지 못하는 시각적 탐색 기능을 제공한다.

### LLM 에이전트의 역할

| 항목 | 설명 |
|------|------|
| **BYOAI** | Claude Code, Codex, Gemini, 오픈소스 모델 등 자유롭게 교체 가능 |
| **활용 시점** | 인제스트, 쿼리, 린트 — 3가지 워크플로우 실행 시 |
| **실행 방식** | 사용자가 LLM 세션에서 자연어로 지시 |
| **읽기 범위** | `raw/` (원시 소스) + `wiki/` (기존 위키) + `wikey.schema.md` (스키마) |
| **쓰기 범위** | `wiki/` 전체 (페이지 생성·수정·삭제, 인덱스·로그 갱신) |
| **쓰기 금지** | `raw/` (불변), `wikey.schema.md` (사용자 승인 없이 수정 불가) |
| **도구 사용** | Obsidian CLI, Git, 셸 명령 (프로바이더별 상세는 각 설정 파일 참조) |

LLM은 **매 세션마다 `wikey.schema.md`를 먼저 읽고** 위키의 구조, 컨벤션, 워크플로우를 파악한 후 작업한다.

### 프로젝트 폴더 = Obsidian 볼트

`~/Project/wikey/` 하나의 폴더가 3가지 역할을 동시에 수행한다:

| 역할 | 근거 |
|------|------|
| **프로젝트 폴더** | LLM 에이전트의 작업 디렉토리 |
| **Obsidian 볼트** | `.obsidian/` 디렉토리 존재 → Obsidian이 인식 |
| **Git 저장소** | `.git/` → 버전 관리 + 원격 백업 |

동기화 문제가 없는 이유는 **원본이 하나**이기 때문이다.

## 디렉토리 구조

```
wikey/
├── wikey.schema.md      # 이 파일 — 프로바이더 독립 마스터 스키마
├── CLAUDE.md            # Claude Code 프로바이더 설정
├── AGENTS.md            # Codex 프로바이더 설정
├── raw/                 # 원시 소스 (불변, LLM은 읽기만)
│   ├── articles/        # 웹 기사, 블로그 포스트
│   ├── papers/          # 논문, 리포트
│   ├── notes/           # 메모, 일기, 팟캐스트 노트
│   └── assets/          # 이미지, 첨부파일
├── wiki/                # LLM이 생성·유지하는 위키
│   ├── index.md         # 콘텐츠 인덱스 (카테고리별)
│   ├── log.md           # 시간순 활동 로그
│   ├── overview.md      # 위키 전체 개요/종합 분석
│   ├── entities/        # 엔티티 페이지 (인물, 조직, 도구 등)
│   ├── concepts/        # 개념 페이지 (이론, 방법론 등)
│   ├── sources/         # 소스별 요약 페이지
│   └── analyses/        # 비교, 분석, 탐색 결과
├── scripts/             # 검증 스크립트
│   ├── validate-wiki.sh # 위키 정합성 검증
│   └── check-pii.sh     # PII 패턴 스캐닝
├── local-llm/           # 로컬 LLM 설정
│   └── system-prompt.md # 로컬 모델용 시스템 프롬프트
└── llm-wiki-kor.md      # 원문 번역 (참고용)
```

## 3계층 아키텍처

| 레이어 | 위치 | 소유자 | 설명 |
|--------|------|--------|------|
| 원시 소스 | `raw/` | 사용자 | 불변. 진실의 원천. LLM은 읽기만 함 |
| 위키 | `wiki/` | LLM | 페이지 생성·업데이트·상호참조 유지 |
| 스키마 | `wikey.schema.md` + 프로바이더 설정 | 사용자+LLM | 구조, 컨벤션, 워크플로우 정의 |

## 시스템 워크플로우 (전체 흐름)

시스템에는 4가지 워크플로우가 있다. 각 워크플로우의 트리거와 데이터 흐름이 다르다.

### 전체 흐름도

```
                         사용자
                     ┌─────┴─────┐
                     │           │
               소스 추가      질문/지시
                     │           │
                     ▼           ▼
              ┌──────────┐  ┌──────────┐
              │  raw/    │  │  LLM    │
              │ (원시소스) │  │  세션    │
              └────┬─────┘  └────┬─────┘
                   │             │
         ┌─────────┼─────────────┼──────────────┐
         │ Phase 1 │             │              │
         │ (수동)  │   사용자가   │              │
         │         │  "인제스트"  │              │
         │         │   지시      │              │
         │ Phase 2 │  fswatch가  │              │
         │ (반자동) │  변경 감지  │              │
         │         │  → 알림 →   │              │
         │         │  사용자 승인 │              │
         └─────────┼─────────────┼──────────────┘
                   │             │
                   ▼             ▼
            ┌─────────────────────────┐
            │      LLM 실행           │
            │  ┌─────┬──────┬──────┐  │
            │  │인제  │쿼리  │린트  │  │
            │  │스트  │      │      │  │
            │  └──┬──┘└──┬──┘└──┬──┘  │
            └─────┼──────┼──────┼─────┘
                  │      │      │
                  ▼      ▼      ▼
            ┌─────────────────────────┐
            │      wiki/ (위키)        │
            │  index.md  ← 매번 갱신   │
            │  log.md    ← 매번 추가   │
            │  sources/  entities/    │
            │  concepts/ analyses/    │
            └───────────┬─────────────┘
                        │
              ┌─────────┼─────────┐
              │         │         │
              ▼         ▼         ▼
          Obsidian   Git 커밋   validate
          (실시간     (이력     -wiki.sh
           렌더링)    추적)    (정합성)
```

### 워크플로우 1: 인제스트 (소스 → 위키)

**트리거**: 사용자가 소스를 추가하고 LLM에게 지시 (Phase 1: 수동, Phase 2: 반자동 알림)

```
사용자: raw/에 소스 추가
  │
  ▼ (Phase 1) 사용자가 직접 지시
  ▼ (Phase 2) fswatch 알림 → 사용자 승인
  │
LLM: 소스 읽기
  │
  ├─► 사용자와 핵심 시사점 논의
  │
  ├─► wiki/sources/source-{name}.md 생성 또는 업데이트 (멱등)
  │
  ├─► wiki/entities/, wiki/concepts/ 페이지 생성 또는 업데이트
  │
  ├─► wiki/index.md 갱신 (새 페이지 등재, 기존 요약 수정)
  │
  ├─► wiki/log.md 에 항목 추가 (날짜, 타입, 영향 페이지, 토큰)
  │
  ├─► wiki/overview.md 갱신 (필요시)
  │
  ├─► validate-wiki.sh 실행
  │     ├─ 통과 → Git 커밋
  │     └─ 실패 → 오류 수정 후 재검증
  │
  └─► Obsidian이 실시간으로 변경 반영 (사용자가 브라우징)
```

**하나의 인제스트가 건드리는 파일**: 10–15개 (소스 요약 1 + 엔티티/개념 N + index + log + overview)

### 워크플로우 2: 쿼리 (질문 → 답변 → 선택적 저장)

**트리거**: 사용자가 LLM 세션에서 질문

```
사용자: "X에 대해 알려줘" / "A와 B를 비교해줘"
  │
LLM: wiki/index.md 읽기
  │
  ├─► 관련 페이지 식별
  │
  ├─► 해당 페이지들 읽기 (entities/, concepts/, sources/)
  │
  ├─► 답변 종합 (인용 포함)
  │
  └─► 가치 있는 답변?
        ├─ 예 → wiki/analyses/에 새 페이지로 저장
        │        → index.md, log.md 갱신
        └─ 아니오 → 대화에서만 답변 (위키 변경 없음)
```

**index.md는 쿼리 시 자동 갱신되지 않는다** — 답변이 위키에 저장될 때만 갱신.

### 워크플로우 3: 린트 (위키 상태 점검)

**트리거**: 사용자가 주기적으로 지시 ("위키 상태 점검해줘")

```
사용자: "린트 실행해줘"
  │
LLM: git diff로 최근 변경 페이지 식별 (증분 린트)
  │   또는 전체 wiki/ 스캔 (전체 린트)
  │
  ├─► 점검 항목:
  │     - 페이지 간 모순
  │     - 새 소스가 대체한 오래된 주장
  │     - 고아 페이지 (인바운드 링크 없음)
  │     - 언급되었지만 페이지 없는 개념
  │     - 누락된 상호참조
  │     - 삭제된 소스에 의존하는 주장
  │
  ├─► 발견 사항을 사용자에게 보고
  │
  ├─► 사용자 승인 후 수정 실행
  │
  └─► log.md에 lint 항목 추가
```

### 워크플로우 4: 소스 삭제/수정 (역방향 전파)

**트리거**: 사용자가 raw/에서 파일 삭제 또는 수정 사실을 LLM에 알림

```
사용자: "raw/articles/old.md 삭제했어. 위키 정리해줘"
  │
LLM: 해당 소스의 wiki/sources/source-{name}.md 확인
  │
  ├─► 해당 소스를 인용하는 모든 엔티티/개념 페이지 검색
  │
  ├─► 각 페이지에서:
  │     - 다른 소스로 뒷받침되는 주장 → 소스 인용만 제거, 내용 유지
  │     - 해당 소스가 유일한 근거 → 내용 제거 또는 "근거 삭제됨" 표시
  │
  ├─► wiki/sources/source-{name}.md 삭제 또는 아카이브
  │
  ├─► index.md, log.md 갱신
  │
  └─► validate-wiki.sh → Git 커밋
```

### 자동화 수준 (Phase별)

| Phase | 인제스트 트리거 | 인덱스/검색 | 린트 |
|-------|---------------|------------|------|
| **Phase 1** (수동) | 사용자가 직접 지시 | index.md 전체 읽기 (LLM이 수동 갱신) | 사용자가 직접 지시 |
| **Phase 2** (반자동) | fswatch 알림 → 사용자 승인 | 카테고리별 인덱스 분할 + qmd 검색 | 주간 스케줄 + 사용자 지시 |

**Phase 2 반자동 흐름:**
```
fswatch가 raw/ 감시
  │
  ├─► 새 파일 감지
  │
  ├─► 사용자에게 알림 (터미널 또는 Obsidian 알림)
  │     "raw/articles/new-article.md 추가됨. 인제스트할까요?"
  │
  ├─► 사용자 승인
  │     ├─ "예" → LLM 인제스트 세션 시작
  │     ├─ "나중에" → 대기 목록에 추가
  │     └─ "무시" → 건너뜀
  │
  └─► 승인된 소스를 인제스트 (워크플로우 1 실행)
```

### 검색/인덱스 확장 전략

위키가 커지면 index.md를 매번 전체 읽는 방식은 **토큰 비용이 기하급수적으로 증가**한다.

핵심 원칙: **검색을 DB에 위임하면 다시 RAG가 된다.** LLM Wiki의 차별점은 LLM이 검색 과정에 참여하는 것이다. **"LLM이 검색의 양쪽 끝에 참여하되, 중간의 대량 스캔만 외부 도구가 담당"**하는 구조로 확장한다.

### LLM 참여형 다층 검색

```
RAG (기존):          DB가 검색 → LLM은 결과만 읽음 (LLM은 수동적)
LLM Wiki (우리 방식): LLM이 쿼리 확장 → 외부 검색 → LLM이 리랭킹 → LLM이 합성
                     (LLM이 검색의 시작과 끝에 참여 = 지능적 다층 검색)
```

| 단계 | 담당 | 역할 | 확장성 |
|------|------|------|--------|
| 1. 쿼리 이해·확장 | **LLM** | 질문 의도 파악, 동의어/변형/가상문서 생성 | 토큰 소량 |
| 2. 후보 수집 | 외부 검색 | BM25 + 벡터로 빠른 1차 필터링 (수천→30개) | O(log n) |
| 3. 리랭킹 | **LLM** | 후보의 실제 관련성 판단 | 30개만 평가 |
| 4. 합성 | **LLM** | 최종 답변 생성, 교차참조, 모순 감지 | 상위 10개만 |

### Phase별 검색 진화

```
Phase 1: LLM이 index.md 전체 읽기 → 직접 판단 → 페이지 읽기
         (LLM이 100% 담당. 단순하지만 <100p에서만 가능)

Phase 2: LLM이 카테고리 인덱스 읽기 + qmd 검색 병행
         (LLM 라우팅 + qmd 보조. 300p까지 가능)

Phase 3: LLM이 쿼리 확장 → qmd가 후보 수집 → LLM이 리랭킹 → 합성
         (qmd MCP 서버 활용. 3,000p까지 가능)

Phase 4: LLM이 쿼리 확장 → 하이브리드 검색(BM25+벡터) → LLM 리랭킹 → 합성
         (qmd 패턴 확장. 10,000p+ 가능)
```

### 참조 프로젝트의 검색 해법

**llmbase**: LLM이 컴팩트한 index.json(제목+요약)을 읽고 **직접 상위 10개 문서를 선택**. DB 없음.

**seCall**: BM25(한국어 형태소 분석) + 벡터 검색 → RRF 융합 + 세션 다양성 필터.

**qmd** (가장 정교): LLM이 쿼리 확장 → BM25+벡터 병렬 검색 → RRF 융합 → **LLM이 리랭킹**. 모든 모델 로컬 실행(<2GB).

```
qmd 파이프라인:
  질문 → LLM 쿼리확장 → [BM25 × N변형 + 벡터 × N변형]
                              │
                              ▼
                         RRF 융합 (상위 30개)
                              │
                              ▼
                         LLM 리랭킹 (상위 10개)
                              │
                              ▼
                         LLM 합성 (최종 답변)
```

### 토큰 절감 효과

| 규모 | 전략 | LLM이 읽는 토큰 | 절감률 | LLM 참여도 |
|------|------|----------------|--------|-----------|
| <100p | index.md 전체 읽기 | ~5,000 | - | 100% |
| 100-300p | 카테고리 분할 + LLM 라우팅 | ~3,750 | 75% | 100% |
| 300-3,000p | LLM 확장 → qmd → LLM 리랭킹 | ~2,500 | 95% | 양끝 참여 |
| 3,000p+ | LLM 확장 → 하이브리드 → LLM 리랭킹 | ~1,500 | 97% | 양끝 참여 |

### Phase 4 보조 메타데이터

DB는 **린트 자동화 + 구조 분석용 보조 도구**로만 사용. 검색의 주체는 LLM.

```sql
-- 린트용 메타데이터 (검색용이 아님)
pages (id, path, title, type, created, updated, source_count)
links (from_page_id, to_page_id)

-- 린트 자동화 예시
-- 고아 페이지: SELECT * FROM pages WHERE id NOT IN (SELECT to_page_id FROM links)
-- 오래된 페이지: SELECT * FROM pages WHERE updated < NOW() - INTERVAL '90 days'
```

## 원시 소스 관리

### 원칙

`raw/`는 **진실의 원천(source of truth)**이다. 위키의 모든 주장은 `raw/`의 소스로 추적 가능해야 한다.

- LLM은 `raw/`를 **읽기만** 한다 (절대 수정하지 않음)
- 사용자만 `raw/`에 파일을 추가·수정·삭제할 수 있다
- `raw/`는 `.gitignore` 대상이다 (PII 보호 — Git에 커밋하지 않음)

### 추가 (Add)

소스를 `raw/`의 적절한 하위 디렉토리에 넣고, LLM에게 인제스트를 지시한다.

| 소스 유형 | 추가 방법 | 저장 위치 |
|----------|----------|----------|
| 웹 기사 | **Obsidian Web Clipper** → 마크다운 변환 후 저장 | `raw/articles/` |
| 웹 기사 (대안) | LLM에 URL 전달 → 내용 읽고 인제스트 | (파일 저장 생략 가능) |
| 논문/PDF | 파일을 직접 복사 또는 드래그앤드롭 | `raw/papers/` |
| 개인 메모 | 직접 작성 또는 다른 앱에서 복사 | `raw/notes/` |
| 이미지 | Obsidian 설정에서 첨부 폴더를 `raw/assets/`로 지정 | `raw/assets/` |
| 회의록 | 음성 → 텍스트 변환(Whisper 등) 후 저장 | `raw/notes/` |
| Slack/이메일 | 텍스트 복사 → 마크다운 파일로 저장 | `raw/notes/` |

**인제스트 지시 예시:**
```
"raw/articles/llm-wiki-karpathy.md를 인제스트해줘"
"raw/papers/ 안의 새 PDF를 모두 인제스트해줘"
"이 URL의 내용을 인제스트해줘: https://example.com/article"
```

### 수정 (Update)

원시 소스는 원칙적으로 **불변(immutable)**이다. 수정이 필요한 경우:

| 상황 | 처리 방법 |
|------|----------|
| 소스 내용이 업데이트됨 | 새 버전을 별도 파일로 추가 → **재인제스트** 지시. LLM이 기존 위키 페이지에 "업데이트됨" 표시 |
| 소스에 오류가 있었음 | 정정 메모를 `raw/notes/`에 추가 → 인제스트. LLM이 영향받는 위키 페이지를 수정 |
| 소스를 잘못 저장함 | 파일 이동 후 LLM에게 알림. 위키 페이지의 소스 경로 갱신 |

**재인제스트 시 LLM의 행동:**
- 기존 `wiki/sources/source-{name}.md`가 있으면 **업데이트** (중복 생성 안 함)
- 변경된 내용을 기존 엔티티/개념 페이지에 반영
- `log.md`에 `re-ingest` 타입으로 기록

### 삭제 (Delete)

소스를 삭제하면 해당 소스에서 파생된 위키 콘텐츠의 신뢰성이 영향받는다.

**삭제 절차:**
1. 사용자가 `raw/`에서 파일 삭제
2. LLM에 삭제 사실을 알림: `"raw/articles/old-article.md를 삭제했어. 위키를 정리해줘"`
3. LLM이 수행하는 작업:
   - `wiki/sources/source-{name}.md`에 삭제 표시 (또는 삭제)
   - 해당 소스만을 근거로 하는 주장이 있는 엔티티/개념 페이지 검토
   - 다른 소스로 뒷받침되는 주장은 유지, 해당 소스 인용만 제거
   - 해당 소스만이 유일한 근거인 주장은 "근거 소스 삭제됨" 표시 또는 해당 내용 제거
   - `index.md` 업데이트
   - `log.md`에 `delete` 타입으로 기록

**PII 삭제 시 주의:** 소스에 PII가 포함되어 있었고 위키에 전파되었다면, 위키 페이지에서도 해당 PII를 제거해야 한다. LLM에게 명시적으로 PII 제거를 지시한다.

### 소스 현황 조회

```bash
# raw/ 내 소스 파일 목록
find raw/ -type f -name "*.md" -o -name "*.pdf" | sort

# 인제스트된 소스 확인 (wiki에 요약 페이지가 있는 소스)
ls wiki/sources/

# 인제스트되지 않은 소스 찾기
# → LLM에게 "인제스트되지 않은 소스가 있는지 확인해줘" 지시
```

## 워크플로우 상세

### 인제스트 (Ingest)

새 소스를 `raw/`에 추가하고 LLM에게 처리를 지시한다.

1. 소스를 읽고 핵심 시사점을 사용자와 논의
2. `wiki/sources/source-{name}.md` 생성 (기존 존재 시 업데이트 — 멱등성 보장)
3. `wiki/index.md` 업데이트
4. 관련 엔티티/개념 페이지 업데이트 또는 생성
5. `wiki/log.md`에 항목 추가 (토큰 사용량 포함)
6. 필요시 `wiki/overview.md` 업데이트
7. `validate-wiki.sh` 실행 → 통과 시 Git 커밋

**대용량 소스** (20페이지+ PDF, 2시간+ 회의록):
- 섹션/챕터/시간 세그먼트별 분할
- 각 청크를 중간 노트로 인제스트
- 합성 패스로 최종 위키 페이지 통합

### 쿼리 (Query)

위키에 대해 질문한다.

1. `wiki/index.md`를 읽어 관련 페이지 파악
2. 관련 페이지를 읽고 답변 종합
3. 인용과 함께 답변 제공
4. 가치 있는 답변은 `wiki/analyses/`에 새 페이지로 저장

### 린트 (Lint)

주기적으로 위키 상태를 점검한다.

- 페이지 간 모순 확인
- 새 소스가 대체한 오래된 주장 확인
- 인바운드 링크 없는 고아 페이지 확인
- 언급되었지만 자체 페이지가 없는 개념 확인
- 누락된 상호 참조 확인
- 삭제된 소스에 의존하는 주장 확인
- 웹 검색으로 채울 수 있는 데이터 공백 제안
- **증분 린트**: git diff 기반으로 변경된 페이지만 점검 (전체 린트는 명시적 요청 시)

## 페이지 컨벤션

### 프론트매터

모든 위키 페이지는 YAML 프론트매터를 포함한다:

```yaml
---
title: 페이지 제목
type: entity | concept | source | analysis
created: 2026-04-10
updated: 2026-04-10
sources: [source1.md, source2.md]
tags: [태그1, 태그2]
---
```

### 네이밍

- 파일명: 소문자, 하이픈 구분 (예: `transformer-architecture.md`)
- 엔티티: 고유명사 그대로 (예: `andrej-karpathy.md`)
- 소스 요약: `source-` 접두사 + 원본 식별자 (예: `source-llm-wiki-gist.md`)

### 상호 참조

- Obsidian 호환 위키링크 사용: `[[페이지명]]` 또는 `[[페이지명|표시텍스트]]`
- 관련 페이지는 "관련 항목" 섹션으로 페이지 하단에 정리

### index.md 형식

```markdown
## 엔티티
- [[entity-name]] — 한 줄 설명 (소스: N개)

## 개념
- [[concept-name]] — 한 줄 설명 (소스: N개)

## 소스
- [[source-name]] — 원본 제목 (날짜)

## 분석
- [[analysis-name]] — 한 줄 설명 (날짜)
```

### log.md 형식

```markdown
## [2026-04-10] ingest | 소스 제목
- 요약 페이지 생성: [[source-name]]
- 업데이트: [[entity-a]], [[concept-b]]
- 신규 페이지: [[entity-c]]

## [2026-04-10] query | 질문 요약
- 답변 저장: [[analysis-name]]

## [2026-04-10] lint | 정기 점검
- 모순 발견: [[page-a]] vs [[page-b]]
- 고아 페이지: [[orphan-page]]
```

## Obsidian CLI 활용 패턴

Obsidian이 실행 중일 때 `obsidian` 명령으로 위키를 조작할 수 있다.

```bash
# 노트 읽기/생성/추가
obsidian read file="overview"
obsidian create name="New Entity" content="# Entity" silent
obsidian append file="log" content="## [2026-04-10] ingest | 새 소스"

# 검색
obsidian search query="검색어" limit=10

# 데일리 노트
obsidian daily:read
obsidian daily:append content="- [ ] 새 태스크"

# 속성(프론트매터) 관리
obsidian property:set name="updated" value="2026-04-10" file="My Note"

# 태그, 백링크
obsidian tags sort=count counts
obsidian backlinks file="overview"
```

자세한 명령어: `obsidian help` 또는 https://help.obsidian.md/cli

## 핵심 원칙

1. **원시 소스는 불변이다** — `raw/`의 파일은 절대 수정하지 않는다
2. **위키는 LLM이 소유한다** — 사용자는 읽고 지시하고, LLM이 작성·유지한다
3. **탐색은 축적된다** — 좋은 쿼리 결과는 위키에 다시 저장한다
4. **인덱스를 항상 최신으로** — 페이지 변경 시 `index.md`를 반드시 업데이트한다
5. **로그는 추가만** — `log.md`는 append-only, 과거 항목을 수정하지 않는다

## LLM Wiki 개인화의 4가지 장점 (Karpathy)

| 원칙 | 설명 |
|------|------|
| 명시성(Explicit) | AI의 지식이 위키로 가시화됨. 무엇을 알고 모르는지 직접 확인·관리 가능 |
| 데이터 소유권(Yours) | 로컬 저장, 특정 업체 종속 없음 |
| 파일 우선(File over app) | 마크다운 등 범용 포맷. Unix 도구·CLI 호환 |
| AI 선택 자유(BYOAI) | Claude, Codex, Gemini 등 자유롭게 연결 |
